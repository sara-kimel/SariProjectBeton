## 1. סקירה כללית וחזון

**הבעיה:** בטון מוכן הוא מוצר מתכלה — לאחר יציקה נותרות לעיתים שאריות במערבל שחייבות להישפך תוך שעות ספורות אחרת הן מתקשות והופכות לפסולת. במקביל, קיימים לקוחות פרטיים/קטנים שצריכים **כמות קטנה** של בטון (גדר, יסוד קטן, רצפה) — כמות קטנה מדי מכדי להצדיק הזמנת מערבל שלם ממפעל.

**הפתרון:** פלטפורמת תיווך שמחברת בזמן אמת בין **קבלן** עם שארית בטון זמינה לבין **לקוחות ממתינים** בקרבת מקום שהצורך שלהם תואם את השארית.

**מודל עסקי (אושר):** תיווך בלבד. המערכת מציגה מחיר ומחברת בין הצדדים; התשלום מתבצע ישירות בין קבלן ללקוח מחוץ למערכת. אין סליקה ב-MVP.

**עקרון מפתח (אושר, תואם לקוד הקיים):**
> הלקוח מזין בקשה ונכנס לרשימת המתנה. **המנוע מופעל רק כאשר קבלן מזין פנייה** — ואז הוא מחפש בקשות לקוחות ממתינות התואמות לפנייה של הקבלן. לקוח שממתין הוא פסיבי עד שמגיעה פנייה תואמת.

## 4. הזרימה המרכזית (End-to-End)

```
לקוח                          מערכת                         קבלן
 │                              │                             │
 │─ הרשמה/התחברות ──────────────►                             │
 │─ יצירת בקשה (מטרה,כמות,מיקום)─► ConcreteRequest(status=OPEN)│
 │                              │  ◄── נכנס לרשימת המתנה       │
 │      (ממתין, פסיבי)          │                             │
 │                              │        ◄─── התחברות ─────────│
 │                              │  ◄─── הזנת פנייה ────────────│  (סוג,כמות,מחיר,תפוגה,מיקום)
 │                              │                             │
 │                              ▼  🔴 המנוע מופעל כאן
 │                        ┌──────────────────────┐
 │                        │ 1. סינון גיאוגרפי     │  (רדיוס, R-tree, Haversine)
 │                        │ 2. סינון לפי מטרה     │  (Purpose של סוג הבטון)
 │                        │ 3. סינון לפי כמות     │  (90%–100% מכמות הפנייה)
 │                        │ 4. דירוג לפי ניקוד    │  (ימי המתנה − זמן נסיעה)
 │                        └──────────┬───────────┘
 │                                   ▼
 │◄── התראה: "נמצאה שארית מתאימה!" ── OfferMatches(status=NOTIFIED)
 │                                   │
 │─ אישור ההצעה ─────────────────────► (הראשון שמאשר זוכה)
 │                                   ▼
 │                        Offer.status=CLOSED, Request.status=CLOSED
 │                        Match.status=ACCEPTED
 │◄─ פרטי הקבלן (טלפון) ──            ──► התראה לקבלן: "לקוח X אישר" + פרטי הלקוח
 │                                   │
 │           לקוחות אחרים שהותאמו ◄── התראה: "ההצעה כבר נתפסה"
```

**נקודות מפתח בזרימה:**
- ✅ (OD-6) הזנת בקשה ע"י לקוח **גם היא מפעילה** את המנוע בכיוון ההפוך — סריקת פניות פתוחות קיימות והתראה מיידית ללקוח אם נמצאה התאמה. המנוע דו-כיווני. (בפועל, כיוון פניית הקבלן נשאר העיקרי, כי הפניות מתכלות.)
- הפעלת המנוע נעשית **סינכרונית** בעת הזנת פנייה, ומחזירה לקבלן את רשימת הלקוחות שהותאמו.
- שליחת ההתראות ואישור הלקוח מתרחשים **א-סינכרונית** לאחר מכן.

## 5. מנוע ההתאמה — לב המערכת

מבוסס על הקוד הקיים ב-`service/matching_engine_service.py` + `service/contractor_matching_controller.py`. להלן האיפיון המחייב (כולל תיקונים לבאגים שאותרו).

### 5.1 קלט וכיווניות
המנוע **דו-כיווני** (OD-6) ורץ בשני טריגרים:
- **טריגר A (עיקרי) — פניית קבלן:** קלט `{ lat, lng, concrete_id, quantity, ... }` → מוצא **בקשות לקוחות** ממתינות ותואמות.
- **טריגר B — בקשת לקוח חדשה:** קלט `{ lat, lng, purpose_id, quantity }` → מוצא **פניות קבלנים** פתוחות (שלא פגו) ותואמות.

שני הכיוונים חולקים את אותם שלבי סינון (גיאו → מטרה/מפרט → כמות) ואת אותה טבלת `OfferMatches`. התיאור להלן מנוסח לטריגר A; טריגר B סימטרי (ראה [5.6](#56-טריגר-הפוך-בקשת-לקוח-חדשה)).

### 5.2 שלבי הצינור (Pipeline)

**שלב 0 — טעינת מועמדים.**
נטענות כל בקשות הלקוחות שסטטוסן **OPEN** בלבד. (תיקון: כיום נטענות *כל* הבקשות ל-R-tree והסינון לפי סטטוס קורה בתוך הלולאה — לא יעיל. יש לסנן `status = OPEN` כבר בשאילתה.)

**שלב 1 — סינון גיאוגרפי.**
- אינדקס מרחבי (R-tree) על (lng, lat) של הבקשות.
- תיבה חוסמת: `delta = radius_meters / 111000` סביב נקודת הקבלן → שליפת מועמדים ראשונית.
- לכל מועמד: חישוב מרחק מדויק ב-**Haversine**; פסילה אם `dist > radius_meters`.
- רדיוס ברירת מחדל: **10,000 מ'**. ✅ (OD-9) הרדיוס ומשקלי הניקוד הם **קבועים בקובץ קונפיג** (ניתנים לכיוונון בקוד, לא חשופים למשתמש).

**שלב 2 — סינון לפי מטרה→מפרט.** ✅ (OD-2)
- הלקוח בוחר **מטרה** בלבד; המערכת **גוזרת ממנה את מפרט הבטון הנדרש** (חוזק/סומך/גודל אבן) לפי מיפוי שמנוהל ע"י המנהל (שדות `req_*` בטבלת `Purpose`).
- נשמרות בקשות שהפנייה של הקבלן **עומדת במפרט הנגזר** של מטרת הבקשה:
  - **חוזק:** `offer.strength.sort_order ≥ req_strength.sort_order` (בטון חזק יותר מתאים גם לדרישה חלשה יותר).
  - **סומך / גודל אבן:** שווים לנדרש אם המטרה מגדירה אותם; אחרת לא מסננים.
- אם למטרה אין מיפוי מפרט — נפילה חיננית ל**התאמה לפי מטרה בלבד** (`request.purpose_id == offer.concrete_type.Purpose_id`), כמו במנוע הקיים.

**שלב 3 — סינון לפי כמות.**
- `min_qty = 0.9 × offer.quantity`, `max_qty = offer.quantity`.
- נשמרות בקשות עם `min_qty ≤ request.quantity ≤ max_qty`.
- כלומר הלקוח צריך כמות שקטנה או שווה לשארית, אך לא פחות מ-90% ממנה (מזעור בזבוז). ✅ **הוכרע (OD-4): נשאר 90%–100%** — המודל הוא שלקוח יחיד לוקח כמעט את כל השארית; לקוח שצריך כמות קטנה בהרבה מהשארית לא יותאם, וזה מכוון.

**שלב 4 — דירוג (Scoring).**
```
waiting_days  = today − request.date            # מס' ימים שהבקשה ממתינה
travel_minutes = (dist_m / 1000) × 2            # הערכה גסה: 2 דק' לק"מ
score = (w1 × waiting_days) − (w2 × travel_minutes)   # w1=5, w2=1
```
מיון יורד לפי `score` — בקשות ותיקות וקרובות מקבלות עדיפות.

### 5.3 פלט
רשימת מועמדים ממוינת, כל אחד עם `{ request_id, customer_id, distance_m, waiting_days, score, ... }`.

### 5.4 מה קורה עם הפלט (חסר כיום — יש לממש)
1. **שמירת הפנייה** ב-DB (כיום `POST /send/` לא שומר כלל!).
2. יצירת רשומות `OfferMatches` לכל מועמד (status=NOTIFIED).
3. שליחת התראות בתוך-אפליקציה ללקוחות שהותאמו, לפי סדר הדירוג. ✅ (OD-5) **broadcast לכל המותאמים בו-זמנית, והראשון שמאשר זוכה** (מתאים למוצר מתכלה בלחץ זמן); היתר מקבלים "כבר נתפסה".
4. החזרת סיכום לקבלן.

### 5.5 שיקולי ביצועים
- ✅ (OD-10) המימוש נשאר **R-tree ב-Python** ל-MVP (פשוט, מתאים לנפח קטן/בינוני). כיום `load_data()` בונה R-tree מחדש בכל ריצה — תקין לנפחים קטנים; להגדלה עתידית לשקול אינדקס מתמיד/קאשינג או `geography`+SP ב-SQL Server (קיים שלד `GetCandidateRequests`).

### 5.6 טריגר הפוך: בקשת לקוח חדשה
כאשר לקוח יוצר בקשה (`POST /concrete-requests/`), לאחר השמירה (status=OPEN) רץ המנוע בכיוון ההפוך (OD-6):
1. **סינון גיאוגרפי** — פניות קבלנים בסטטוס OPEN ושלא פג `expiry_time`, בתוך הרדיוס ממיקום הבקשה.
2. **מטרה→מפרט** — הפנייה עומדת במפרט הנגזר ממטרת הבקשה (כמו שלב 2).
3. **כמות** — `request.qty ∈ [0.9 × offer.qty, offer.qty]` (אותו כלל, לכל פנייה).
4. לכל פנייה תואמת נוצרת רשומת `OfferMatches` (NOTIFIED) ונשלחת התראה **ללקוח** ("נמצאה פנייה מתאימה"); הלקוח יכול לאשר מיד.

הערה: זהו הכיוון המשלים — הוא מכסה מצב שבו כבר קיימות פניות פתוחות כשהלקוח נכנס. בפועל, בגלל שהפניות מתכלות (תפוגה בשעות), הטריגר העיקרי יישאר פניית הקבלן.

## 6. מכונות מצב (Lifecycle)

### 6.1 בקשת לקוח (`ConcreteRequests.status`)
```
OPEN ──(לקוח אישר פנייה)──► CLOSED
  │
  └──(הלקוח ביטל)──────────► CANCELLED
```
- **OPEN** — ממתין, זמין להתאמה. ברירת מחדל ביצירה.
- **CLOSED** — הלקוח אישר פנייה ונסגרה עסקה.
- **CANCELLED** — הלקוח ביטל ידנית.
- ✅ (OD-7) **אין תפוגה אוטומטית לבקשה** — היא ממתינה עד שנסגרת (אישור) או מבוטלת.

> תיקון קונספטואלי: כיום המנוע מזהה "פתוח" לפי `status IS NULL`. יש להחליף לשדה סטטוס מפורש עם ברירת מחדל `OPEN`, והמנוע יסנן `status = OPEN`.

### 6.2 פניית קבלן (`ContractorConcreteRequests.status` — עמודה חדשה)
```
OPEN ──(לקוח כלשהו אישר)──► CLOSED
  │
  ├──(פג expiry_time)──────► EXPIRED
  └──(הקבלן ביטל)──────────► CANCELLED
```

### 6.3 התאמה (`OfferMatches.status` — טבלה חדשה)
```
NOTIFIED ──(הלקוח אישר)────► ACCEPTED
   │
   ├──(הלקוח דחה)───────────► DECLINED
   ├──(לקוח אחר אישר קודם)──► SUPERSEDED
   └──(הפנייה פגה)──────────► EXPIRED
```

## 7. מודל הנתונים (DB)

מסד: **SQL Server**, DB בשם `beton`. הסכימה הקיימת ב-`server/db/schema.sql` (9 טבלאות). להלן הטבלאות + **השינויים הנדרשים** (מסומנים 🔧 = שינוי, ➕ = חדש).

### 7.1 טבלאות קיימות (עם תיקונים)

**Customers** (id החל מ-100)
| עמודה | טיפוס | הערות |
|-------|-------|-------|
| id | INT IDENTITY PK | |
| first_name, last_name | NVARCHAR | |
| user_name | NVARCHAR(50) 🔧 | **UNIQUE** 🔧, נדרש לאימות |
| password | — | 🔧 **להסיר**; להחליף ל-`password_hash` |
| password_hash ➕ | NVARCHAR(255) | bcrypt/argon2 |
| phone | NVARCHAR(20) | |
| created_at ➕ | DATETIME2 DEFAULT SYSDATETIME() | |

**Contractors** (id החל מ-300) — אותם שינויים כמו Customers (user_name UNIQUE, password_hash, created_at).

➕ **Admins** — טבלת מנהלים (או `is_admin` על טבלה מאוחדת; ראה [OD-8](#16-החלטות-פתוחות-לאישור)).

**טבלאות lookup** — `Strength` (1100), `Reliant` (1200), `Stone_size` (1300), `Purpose` (1400): **חובה לזרוע נתונים** (seed) — כיום ריקות, ולכן המנוע לא ימצא התאמות. ➕ seed script. שינויים ל-OD-2 (מטרה→מפרט):
- `Strength` ➕ `sort_order INT` — לסדר חוזקים להשוואת `≥` (B-20 < B-30 < B-40).
- `Purpose` ➕ `req_strength_id`, `req_reliant_id`, `req_stone_size_id` (FK nullable) — המפרט הנגזר לכל מטרה, מנוהל ע"י המנהל. אם NULL → נפילה להתאמה לפי מטרה בלבד.

**Concrete_type** (id החל מ-2000) — צירוף `strength_id + Reliant_id + Stone_size_id + Purpose_id`. ללא שינוי מבני; דורש seed + ניהול ע"י מנהל.

## 9. ארכיטקטורת צד שרת

השכבות הקיימות טובות ויש לשמר אותן:
```
controller/  →  service/  →  repository/  →  models/ (SQLAlchemy)
     ▲             ▲              ▲
    dto/        (לוגיקה)      (גישת DB)
```
- **controller** — הגדרת endpoints, ולידציה, קריאה ל-service/repository.
- **service** — לוגיקה עסקית (מנוע התאמה, תמחור, התראות, אישור עסקה טרנזקציוני).
- **repository** — CRUD מול ה-DB.
- **dto** — Pydantic, קלט/פלט.
- **models** — SQLAlchemy ORM.

**עקרונות:**
- כל פעולה שמשנה כמה טבלאות (אישור עסקה) — בתוך **טרנזקציה** אחת.
- שירותים חיצוניים (Geocoding) מאחורי interface (כבר קיים שלד `RegionService`).
- מנוע ההתאמה כ-service נקי, בלי תלות ב-FastAPI.

## 10. אבטחה, אימות והרשאות

**אושר: שם משתמש + סיסמה, עם תפקידים.**

- **סיסמאות:** לעולם לא בטקסט גלוי. Hash עם bcrypt/argon2. (תיקון קריטי — כיום גלוי, 15 תווים.)
- **טוקנים:** JWT עם `sub` (מזהה) + `role`. תוקף + refresh (או תוקף ארוך ל-MVP).
- **הרשאות (RBAC):**
  - לקוח: רק הבקשות/ההתאמות/ההתראות שלו.
  - קבלן: רק הפניות/ההתאמות/ההתראות שלו.
  - מנהל: ניהול lookups + צפייה כוללת.
- **Middleware:** dependency ב-FastAPI שמאמת JWT ומזריק `current_user`; dependency נוסף לבדיקת role.
- **בעלות (Ownership):** בכל endpoint שמקבל `{id}` — לוודא שהמשאב שייך למשתמש המחובר (מניעת IDOR).
- **CORS:** בייצור להגביל ל-origin של הלקוח בלבד (כיום `*`).
- **ולידציה:** קואורדינטות בטווח חוקי, כמות > 0, מחיר תקין, expiry עתידי.
- **Rate limiting** על login והרשמה (הגנה מפני brute-force) — שלב מאוחר.
- ✅ (OD-12) **שחזור סיסמה:** ל-MVP **איפוס ע"י מנהל** (אין מייל/SMS). המנהל מאפס לסיסמה זמנית והמשתמש מחליף בכניסה הבאה.

## 11. מערכת ההתראות

**אושר: התראות בתוך האפליקציה בלבד (In-app), בלי SMS/ווטסאפ/מייל ב-MVP.**

- כל אירוע יוצר רשומת `Notifications` למשתמש הרלוונטי.
- הלקוח רואה מונה לא-נקראו + מרכז התראות; רענון ב-polling (למשל כל 15–30 שנ') או WebSocket בשלב מתקדם.
- **אירועים מחוללי התראה:**
  | אירוע | אל | תוכן |
  |-------|-----|------|
  | נמצאה התאמה לבקשה | לקוח | "נמצאה שארית בטון מתאימה! מחיר X, מרחק Y ק"מ. לאישור..." |
  | לקוח אישר פנייה | קבלן | "לקוח אישר את הפנייה שלך. טלפון: ..." |
  | הפנייה נתפסה ע"י אחר | לקוחות שהותאמו | "ההצעה כבר נתפסה" |
  | פנייה עומדת לפוג | קבלן | "הפנייה שלך תפוג בעוד X" (שלב מתקדם, דורש job) |
- ✅ (OD-11) **תפוגות בבדיקה עצלה (lazy):** פנייה שעבר `expiry_time` שלה מטופלת כ-EXPIRED בזמן שהיא נטענת/נשאלת (וה-Matches שלה → EXPIRED). אין job רקע ואין התראת תפוגה יזומה ל-MVP; לכן השורה "פנייה עומדת לפוג" בטבלה למעלה נדחית לשלב מאוחר.

## 13. מקרי קצה — מקיף

### 13.1 בקשות לקוח
- בקשה ללא `lat/lng` → נדחית (חובה למנוע).
- כמות ≤ 0 / ריקה / חורגת מ-DECIMAL(6,2) → ולידציה.
- מטרה שלא קיימת ב-Purpose → 400.
- לקוח מגיש כמה בקשות במקביל → מותר; כל אחת מטופלת עצמאית.
- עריכת בקשה שכבר הותאמה/נסגרה → חסום (409) או מותר רק ב-OPEN.
- ביטול בקשה שיש לה התאמות NOTIFIED פעילות → יש לבטל את ההתאמות + להודיע.
- טבלת Purpose ריקה → אף בקשה לא תותאם לעולם (כשל שקט!) → יש להזהיר/לזרוע seed.

### 13.2 פניות קבלן ומנוע
- אין בקשות ממתינות בכלל → המנוע מחזיר רשימה ריקה; להודיע לקבלן "אין לקוחות מתאימים כרגע"; הפנייה נשמרת ונשארת OPEN עד תפוגה.
- כל המועמדים מחוץ לרדיוס → רשימה ריקה.
- `concrete_id` ללא `Purpose_id` (NULL ב-Concrete_type) → שלב 2 מחזיר ריק; לטפל בחן.
- כמות פנייה קטנה מ-90% מכל בקשה → אף התאמה (הכלל מחמיר — ראה OD-4).
- `expiry_time` בעבר בעת הזנה → ולידציה (חייב עתידי).
- שתי פניות של אותו קבלן שמתאימות לאותה בקשה → הבקשה יכולה לקבל שתי התראות; אישור אחת סוגר את הבקשה ומבטל את השנייה.
- `lat/lng` לא תקינים/מחוץ לישראל → ולידציה/אזהרה.
- כמות עם יותר מ-2 ספרות עשרוניות → עיגול/דחייה.

### 13.3 אישור ומרוץ (Concurrency)
- **שני לקוחות מאשרים אותה פנייה במקביל** → חובה עדכון אטומי: `UPDATE Offer SET status=CLOSED WHERE id=? AND status=OPEN` — רק ה-UPDATE שהשפיע על שורה זוכה; השני מקבל 409 "כבר נתפסה".
- לקוח מאשר פנייה שכבר פגה (`expiry_time` עבר) → 409/410 "ההצעה פגה".
- לקוח מאשר פנייה שכבר CLOSED → 409.
- לקוח מאשר בעוד הבקשה שלו כבר CLOSED (אישר פנייה אחרת קודם) → חסום; בקשה יכולה לאשר רק פנייה אחת.
- לאחר אישור: לעדכן במחזור אחד — Offer=CLOSED, Request=CLOSED, Match=ACCEPTED, שאר ה-Matches של אותה פנייה=SUPERSEDED, שאר ה-Matches של אותה בקשה=מבוטלות; ולשלוח התראות.

### 13.4 תפוגות וזמן
- פנייה פגה לפני שמישהו אישר → status=EXPIRED, כל ה-Matches=EXPIRED, הבקשות חוזרות למעשה ל-OPEN (הן ממילא נשארו OPEN).
- שעון שרת מול לקוח, אזורי זמן → לעבוד ב-UTC ולהמיר בתצוגה.
- `date` הוא DATE (רזולוציית יום) → דירוג "ימי המתנה" ברזולוציית יום; לשקול DATETIME אם צריך דיוק.

### 13.5 אימות והרשאות
- לקוח ניגש ל-endpoint של קבלן → 403.
- גישה למשאב של משתמש אחר (IDOR) → 403.
- טוקן פג/לא תקין → 401.
- שם משתמש כבר קיים בהרשמה → 409.
- סיסמה חלשה → ולידציה (מדיניות בסיסית).

### 13.6 נתונים ומצב
- מחיקת לקוח/קבלן עם בקשות/פניות פעילות → למנוע או ל-soft-delete (שלמות FK).
- מחיקת ערך lookup בשימוש → למנוע (FK).
- שדה `region` הלא-עקבי → להסיר/לגזור (ראה OD-3).
- קונפליקט טיפוס `status` (DECIMAL מול NVARCHAR מול NULL) → להאחיד לפי מכונת המצב.

### 13.7 גיאוגרפיה/מפה
- טעינת Google Maps — **נפתרה (2026-07-03):** המפה נטענת ועובדת (billing מופעל על מפתח תקין ב-`client/.env`). אין צורך בחלופה; לכל היותר להציג הודעה ידידותית אם אי-פעם תיכשל הטעינה.
- לחיצה מחוץ לגבולות הגיוניים → אזהרה.

## 17. המלצות נוספות (מה עוד כדאי להוסיף)

מעבר למה שכבר באיפיון, אלה תוספות שלדעתי חשובות למוצר אמיתי. מסומנות לפי עדיפות: **[חובה]** קריטי לתפקוד, **[מומלץ]** משפר משמעותית, **[עתידי]** לשלב מאוחר.

### 17.1 מוצר ואמון

- **[הוכרע — נדחה ל-post-MVP] הגבלת ההתראות בתוך-אפליקציה.** בטון הוא מוצר בלחץ זמן (תפוגה בשעות), ולקוח שאינו פתוח באפליקציה ברגע ההתאמה עלול לפספס. הוחלט ([OD-13](#16-החלטות-פתוחות-לאישור)): ל-MVP **התראות בתוך-אפליקציה בלבד**, המגבלה מוכרת ומקובלת. אם יתברר בפועל שלקוחות מפספסים התאמות — התוספת הראשונה שנוסיף היא **Web Push** לאירוע "נמצאה התאמה" (חינמי, עובד גם כשהדפדפן סגור), ורק אז לשקול SMS.
- **[מומלץ] חשיפת פרטי קשר רק לאחר התאמה.** טלפון של הצד השני נחשף רק אחרי אישור עסקה — מונע גרידה (scraping) של מספרים ושומר על פרטיות.
- **[מומלץ] דירוג ומוניטין.** דירוג 1–5 + הערה לאחר עסקה שנסגרה, לשני הכיוונים. בונה אמון בין זרים שמעבירים כסף/סחורה, ומאפשר לסנן משתמשים בעייתיים.
- **[מומלץ] דיווח וחסימה.** דיווח על אי-הגעה/התנהגות, וחסימת משתמש.
- **[עתידי] אימות קבלנים.** תג "מאומת" (ח.פ / רישיון), מעלה אמון ואיכות.
- **[עתידי] תמונות לפנייה.** הקבלן יכול לצרף תמונה של הבטון/האתר — מגביר ביטחון בלקוח.
- **[חובה] תנאי שימוש ומדיניות פרטיות.** מכיוון שמשתפים מספרי טלפון — נדרשת הסכמה מפורשת + עמוד תנאים.

### 17.2 חוויית משתמש ונגישות

- **[חובה] Mobile-first / PWA.** הקבלן בשטח עם טלפון. אפליקציה שניתן "להתקין" (PWA) עם אייקון וריצה במסך מלא. PWA גם תנאי מקדים ל-Web Push.
- **[מומלץ] השלמת כתובת אוטומטית + Reverse Geocoding.** Google Places Autocomplete לשדה הכתובת, ומילוי אוטומטי של הכתובת מלחיצה על המפה (ולהפך) — מפחית שגיאות מיקום.
- **[מומלץ] ספירה לאחור חיה לתפוגה** בכרטיס הפנייה/ההתאמה, ועדכון חי של סטטוס.
- **[מומלץ] מערכת עיצוב עקבית** (Toasts, Skeletons, Empty/Error states, StatusBadge) — מוגדר כבר בסעיף 12.6.
- **[עתידי] ריבוי שפות (i18n).** עברית עכשיו; לשוק הישראלי לשקול ערבית/רוסית בהמשך — לבנות מוכן ל-i18n מראש.

### 17.3 טכני וארכיטקטורה

- **[מומלץ] עדכוני זמן אמת.** ל-MVP polling להתראות/סטטוס; בהמשך WebSocket/SSE לחוויה מיידית (קריטי ללחץ הזמן).

- **[חובה] Idempotency ומניעת כפילויות.** מפתח idempotency על יצירת פנייה ועל אישור, למניעת שליחה כפולה/מרוץ.
- **[מומלץ] Pagination/סינון/מיון** בכל מסכי הרשימה (בקשות, פניות, התראות, משתמשים).
- **[חובה] מיגרציות DB מסודרות (Alembic).** במקום עריכה ידנית של `schema.sql` — לנהל שינויי סכימה בגרסאות.
- **[מומלץ] בדיקות אוטומטיות.** יחידה למנוע ההתאמה (הלוגיקה הקריטית: גיאו/מטרה/כמות/ניקוד) ואינטגרציה לתרחיש המרוץ (שני לקוחות מאשרים במקביל).
- **[מומלץ] טיפול שגיאות ולוגים אחידים** בצד שרת (כבר יש logger בסיסי) + מזהה בקשה (request id).
- **[עתידי] קאשינג/אינדקס מתמיד** למנוע ההתאמה בקנה מידה (ראה OD-10).

### 17.4 תפעול, אבטחה ותאימות

- **[חובה] Hashing לסיסמאות + JWT** (כבר בסעיף 10) — התיקון הכי דחוף מבחינת אבטחה.
- **[מומלץ] Rate limiting + הגנת brute-force** על login/register.
- **[מומלץ] מחיקה רכה (Soft delete) ויומן פעילות (Audit log)** — לשמירת שלמות היסטוריה ועסקאות.

- **[עתידי] גיבויים אוטומטיים ל-DB וניטור (health/uptime).**

>
> סעיף [14](#14-פערים-מהמצב-הקיים-ורשימת-ניקוי) מתאר את מצב ה**פרה**-MVP; סעיף זה מתאר את מה שנותר לתקן **אחרי** שה-MVP נבנה. סטטוס פריט: ⬜ מתוכנן · 🔧 בתהליך · ✅ בוצע.

### 18.1 תיקונים קריטיים — חובה לפני עלייה לאוויר (P0)

> ✅ **בוצע (2026-07-03): כל 6 תיקוני ה-P0 יושמו ואומתו — 52 בדיקות שרת ירוקות, כולל 3 בדיקות רגרסיה חדשות ל-FIX-2/FIX-3.**

| מזהה | תיאור קצר | קובץ עיקרי | חומרה | סטטוס |
|---|---|---|---|---|
| FIX-1 | הרשאות: `/customers` ו-`/contractors` פתוחים לכל משתמש מאומת — IDOR + דליפת טלפונים המונית | `controller/customer_controller.py`, `controller/contractor_controller.py` | קריטי | ✅ |
| FIX-2 | מרוץ אישור: אין שער אטומי על הבקשה → double-booking (שני קבלנים מקבלים את פרטי הלקוח) | `service/deal_service.py` | גבוה | ✅ |
| FIX-3 | ביטול בקשה/פנייה = מחיקה קשה → קורס על FK של OfferMatches (500); `CANCELLED` לא ממומש | `repository/concrete_request_repository.py` + בקרים | גבוה | ✅ |
| FIX-4 | מפתח JWT ברירת-מחדל לא בטוח, נטען בשקט → זיוף טוקן admin | `config.py`, `.env.example` | גבוה | ✅ |
| FIX-5 | `DB_SERVER` מקובע ל-`localhost` ומתעלם ממשתנה הסביבה | `config.py` | בינוני | ✅ |
| FIX-6 | קדם-סינון R-tree לא מתוקן ל-`cos(lat)` → התאמות תקפות נופלות בקצה מזרח/מערב | `service/matching_engine_service.py` | בינוני | ✅ |

**מה ייעשה בכל פריט:**

- **FIX-1** — רשימת-הכל וה-`POST` יוגבלו ל-`admin` בלבד; `GET/PUT/DELETE /{id}` יותרו רק לבעלים-או-admin (בדיוק כדפוס `_owns_or_admin` הקיים כבר ב-`concrete_request_controller`); שדה `phone` יוסתר מה-Response למי שאינו בעלים/מנהל/צד-לעסקה-סגורה. **מתקן חור אבטחה שמנטרל את מודל הסתרת פרטי-הקשר של §17.1 וסותר §10/§13.5.**
- **FIX-2** — יתווסף שער אטומי סימטרי על הבקשה: `UPDATE ConcreteRequests SET status='CLOSED' WHERE request_id=:rid AND status='OPEN'` עם בדיקת `rowcount==0 → rollback + 409`; עדכון ה-`ACCEPTED` יותנה ב-`AND status='NOTIFIED'` (הגנת-עומק), תוך שמירת סדר הנעילה פנייה→בקשה למניעת deadlock.
- **FIX-3** — המחיקה הקשה תוחלף במעבר סטטוס ל-`CANCELLED` (soft delete), ובאותה טרנזקציה יסומנו ההתאמות ה-`NOTIFIED` של אותה בקשה/פנייה כ-`SUPERSEDED` + תישלח התראת "בוטל" לצד השני. כך נאכפות מכונות המצב §6.1/§6.2 ונמנעת שגיאת ה-FK 547.
- **FIX-4** — תתווסף בדיקת אתחול (config/startup) שתעלה חריגה אם `JWT_SECRET` ריק / שווה לברירת-המחדל / קצר מדי, כשלא במצב פיתוח מפורש; הערך המפורש יוסר מ-`.env.example`.
- **FIX-5** — החזרת הקריאה למשתנה סביבה: `DB_SERVER: str = os.getenv("DB_SERVER", "localhost")`, כדי שההגדרה ב-`.env` תפעל כמתועד.
- **FIX-6** — חישוב `delta_lng = radius_meters / (111000 * cos(radians(lat)))` בנפרד מ-`delta_lat` (עם הגנה מ-`cos≈0`); סינון ה-Haversine המדויק נשאר כפי שהוא, כך שתיבת ה-R-tree תהיה על-קבוצה אמיתית של מעגל הרדיוס.

### 18.2 הקשחות ואיכות (P1)

- **ניקוי קוד מת ומלכודות** — מחיקת `service/contractor_concrete_service.py` + `service/price_service.py` (⚠️ מכפילים מחיר ×2 בסתירה ל-§14 — מוקש רדום), `service/candidate_selector.py`, `service/region_service.py`, ו-`utils.py` (ריק); שינוי-שם/מיזוג של `service/contractor_matching_controller.py` (מכיל רק `TestService` — שם מטעה).
- **טיפול שגיאות אחיד (§8.7)** — הוספת `exception_handler` ל-`Exception` (500 → `{"detail"}` בעברית) ול-`RequestValidationError` (422 → מחרוזת עברית מגובשת) ב-`app.py`; שקילת `request-id` בלוגים.
- **שכבתיות (§9)** — העברת שאילתות ה-DB הישירות מ-`admin_controller` ל-repository/שירות סטטיסטיקה, ושימוש ב-`set_password_hash` במקום `commit` ישיר בתוך `auth_controller`.
- **הקשחת DB** — העתקת כל אינדקסי מיגרציה `005` ל-`schema.sql`; הוספת `CHECK` על עמודות `status` (אכיפת ה-enum ברמת ה-DB); `customer_id`/`purpose_id` → `NOT NULL`; הפיכת מיגרציה `002` לאידמפוטנטית אמיתית; `DBCC CHECKIDENT(... , RESEED)` ב-`seed.py` לשימור טווחי ה-IDENTITY.
- **אבטחה נוספת** — ביטול טוקנים בשינוי/איפוס סיסמה (`token_version`/`password_changed_at` בבדיקת ה-decode); הקשחת rate-limit (Redis או דרך ה-proxy, קריאת `X-Forwarded-For` מהימנה, הגבלת גודל המילון); קריאת סיסמת ה-admin מ-env (במקום קבוע ב-`seed.py`); צמצום enumeration של שמות משתמש; עטיפת ה-`insert` ל-`OfferMatches` ב-savepoint מול אילוץ ה-UNIQUE.
- **לקוח** — הפרדת מצב "שגיאה" מ-"ריק" בלוחות הבקרה (במקום להציג 0 שקט); `role="alert"`/`aria-live` להודעות (נגישות); החלפת ה-redirect הקשיח ב-401 בניווט SPA + שקילת refresh-token; רענון מונה פעמון ההתראות ב-focus ומיד לאחר קריאה; תיעוד סיכון אחסון הטוקן ב-`localStorage`.
