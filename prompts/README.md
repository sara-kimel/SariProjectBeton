# פרויקט שרי — פרומפטים לפיתוח מדורג

תיקייה זו מכילה **פרומפט עצמאי לכל שלב פיתוח**. כל קובץ שלב נבנה כדי להיפתח **בצ'אט חדש**: הוא מכיל את כל ההקשר, את המשימות חוצות DB→שרת→לקוח, ובדיקות מקצה-לקצה.

## איך משתמשים
1. פותחים צ'אט חדש של Claude Code **בתיקיית הפרויקט**.
2. כותבים: `בצע את השלב לפי prompts/phase-N-....md` (או מדביקים את תוכן הקובץ).
3. הסוכן יקרא את `SPEC.md` (האיפיון המלא) ואת קובץ ה-README הזה, ואז יבצע את השלב.
4. מבצעים את השלבים **לפי הסדר** (0 → 6). כל שלב מסתמך על קודמו.
5. בסוף כל שלב: מריצים את בדיקת ה-E2E, ומעדכנים את "הגדרת סיום" (למטה).

## הקשר מאסטר (כל שלב מניח את זה)

### מה זה הפרויקט
**"Beton" / פרויקט שרי** — פלטפורמת תיווך שמחברת **קבלנים** עם שאריות בטון מתכלות (מתקשות תוך שעות) ל**לקוחות** שצריכים כמות קטנה של בטון. **תיווך בלבד** — מציגים מחיר ומחברים בין הצדדים, בלי סליקה.

**הזרימה:** לקוח מזין בקשה ← נכנס לרשימת המתנה (סטטוס OPEN). קבלן מזין פנייה ← **מפעיל את מנוע ההתאמה** שמוצא בקשות תואמות (גיאו → מטרה→מפרט → כמות → ניקוד) ← לקוחות מותאמים מקבלים התראה ← הראשון שמאשר זוכה והעסקה נסגרת. המנוע דו-כיווני: גם בקשת לקוח חדשה סורקת פניות פתוחות.

> **מסמך האמת:** `SPEC.md` בשורש הפרויקט (16 סעיפים). תמיד לקרוא אותו לפני עבודה. גרסה קריאה: `SPEC.html`.

### החלטות שננעלו (OD-1..OD-13)
- **אימות:** שם משתמש + סיסמה **מוצפנת (hash)**, תפקידים לקוח/קבלן/מנהל, JWT + RBAC. **תפקיד יחיד למשתמש**; טבלאות נפרדות Customers/Contractors + Admins.
- **התראות:** בתוך-אפליקציה בלבד (טבלת Notifications + polling). בלי SMS/ווטסאפ/מייל.
- **מנוע:** R-tree ב-Python. רדיוס 10 ק"מ ומשקלים (w1=5, w2=1) קבועים בקונפיג.
- **התאמה:** מטרה→מפרט — הלקוח בוחר מטרה, המערכת גוזרת מפרט נדרש (חוזק≥/סומך/גודל אבן) לפי מיפוי בטבלת Purpose.
- **כמות:** `request.qty ∈ [0.9×offer.qty, offer.qty]` (לקוח יחיד לוקח כמעט את כל השארית).
- **אסטרטגיית התראה:** broadcast לכל המותאמים, הראשון שמאשר זוכה.
- **תפוגות:** פניית קבלן — תפוגה (`expiry_time`) בבדיקה עצלה (lazy) בזמן קריאה. בקשת לקוח — **ללא תפוגה**.
- **שחזור סיסמה:** איפוס ע"י מנהל (אין ערוץ חיצוני).
- **region:** להסיר מה-DTO.

### מחסנית טכנולוגית והרצה
- **שרת:** FastAPI + SQLAlchemy 2 + pyodbc, ב-`server/`. הרצה: `uvicorn app:app --reload --port 8001` (מתוך `server/`). מסמכי API: `http://localhost:8001/docs`.
- **DB:** SQL Server מקומי, DB בשם `beton`, **Windows Authentication**, דרייבר `ODBC Driver 17 for SQL Server`. הגדרות ב-`server/config.py` / `server/.env`. סכימה ב-`server/db/schema.sql`.
- **לקוח:** Vite 8 + React 19 + TS, ב-`client/`. הרצה: `npm install` ואז `npm run dev` (פורט 5173). בדיקת build: `npm run build`.
- **Node:** v22 (מותקן). **Python:** ה-`python`/`py` שב-PATH הם stub של חנות Microsoft — **יש לאתר/ליצור סביבת Python אמיתית** (venv) ולהתקין `requirements.txt` לפני הרצת השרת.
- **מפתח Google Maps:** ב-`client/.env` (`VITE_GOOGLE_MAPS_API_KEY`). הסביבה מאחורי **NetFree** (יירוט SSL) — עלול להשפיע על טעינת Maps בדפדפן.

### מצב נוכחי — מה עובד ומה חסר/שבור
**עובד:** CRUD בסיסי לכל הישויות (controllers/repositories/DTOs), חיבור DB, המנוע הגיאוגרפי (R-tree+Haversine+ניקוד) + סינון מטרה + סינון כמות, שלד לקוח (טופס בקשה + מפה).

**שבור/חסר (עיקרי):**
- `POST /contractor-offers/send/` **לא שומר** את הפנייה ולא יוצר התאמות/התראות — רק מסנן ומחזיר.
- `POST /contractor-offers/` (יצירה) **מבוטל** בקוד (מוער).
- **אין אימות** — סיסמאות בטקסט גלוי.
- **אין** זרימת אישור לקוח, **אין** טבלאות OfferMatches/Notifications.
- **`status`** לא עקבי (מודל DECIMAL NOT NULL / DB nullable / DTO string).
- טבלאות ה-lookup **ריקות** (Purpose וכו') → המנוע לא ימצא כלום.
- `rtree` (בשימוש המנוע) **חסר מ-`requirements.txt`** (דורש גם `libspatialindex`).
- קבצי זבל/ניסוי ב-`server/service/` וב-`server/` (ראה שלב 0).

### מוסכמות עבודה
- **בונים על הקיים, לא כותבים מחדש.** שומרים על הארכיטקטורה השכבתית (controller→service→repository→models, dto בצד).
- **RTL עברית** בכל ה-UI. מטבע ₪, כמות מ"ק, מרחק ק"מ.
- כל שינוי חוצה-טבלאות (אישור עסקה) בתוך **טרנזקציה** אחת.
- שמות עמודות/טבלאות קיימים נשמרים (יש חוסר-אחידות באותיות גדולות, למשל `Reliant`, `Stone_size`, `Purpose` — לא לשבור).
- לא לחשוף סודות בקוד; CORS מוגבל בייצור.

### גישת בדיקות (נבנית בשלב 0)
- **שרת:** `pytest` + FastAPI `TestClient`. תיקייה `server/tests/`.
- **E2E חוצה-מערכת:** תרחיש שמריץ רצף קריאות API אמיתיות מול השרת החי (סקריפט תחת `server/tests/e2e/`), + **אימות ידני ב-UI** לפי צעדים שמפורטים בכל שלב.
- כל שלב מוסיף בדיקות משלו; בסוף כל שלב **כל הסוויטה חייבת לעבור**.

## הגדרת סיום (בכל שלב)
1. כל משימות ה-DB/שרת/לקוח בוצעו לפי הקובץ.
2. **בדיקת ה-E2E של השלב עברה**, וגם `npm run build` בלקוח.
3. עדכון `SPEC.md` (סעיף 15 — סימון השלב כבוצע) אם רלוונטי.
4. עדכון הזיכרון (`memory/project-sari-state.md`) עם מה שנבנה והחלטות חדשות.
5. סיכום קצר למשתמש: מה נבנה, איך נבדק, ומה השלב הבא.
